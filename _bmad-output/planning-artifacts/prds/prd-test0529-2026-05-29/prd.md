# PRD: 중고거래 모바일 플랫폼 MVP

## 0. 문서 목적

이 문서는 위치 기반 중고거래 모바일 앱 MVP의 제품 요구사항을 정의한다. 독자는 개발자, 아키텍트, UX 디자이너이며, 이 PRD는 하위 아키텍처 설계·에픽·스토리 작성의 기준 문서로 사용된다. 기능 요구사항(FR)은 전역 번호로 안정적으로 참조되며, 가정은 `[ASSUMPTION]` 태그로 인라인 표기하고 §10에 모았다. UX 문서 및 아키텍처 설계는 이 PRD 이후 단계에서 별도 작성된다.

**기술 스택 요약:** 모바일 앱(Flutter) + REST/WebSocket API 서버(FastAPI + PostgreSQL) + 운영자 어드민 패널(Next.js).

---

## 1. 비전

이 앱은 같은 동네에 사는 사람들이 안 쓰는 물건을 서로 사고팔 수 있는 모바일 플랫폼이다. 판매자는 사진 몇 장과 짧은 설명만으로 수분 안에 상품을 등록하고, 구매자는 동네 피드를 훑다가 마음에 드는 물건을 발견해 곧바로 채팅으로 연락할 수 있다.

기존 중고거래 플랫폼(당근마켓)의 핵심 가치 — 동네 기반 신뢰 거래, 빠른 직거래, 간편한 채팅 — 를 Flutter 모바일 앱 + FastAPI 백엔드 + PostgreSQL 스택으로 직접 구현하는 것이 이 MVP의 목표다. 결제 시스템과 정밀 위치 인증 없이도 핵심 거래 흐름이 동작한다는 것을 검증한다. 운영자용 어드민 패널은 Next.js로 별도 제공한다.

## 2. 타겟 사용자

### 2.1 해결하는 일 (Jobs To Be Done)

- **판매자**: 집에 잠들어 있는 물건을 빠르게 올려 동네 이웃에게 팔고 싶다.
- **구매자**: 새 제품보다 저렴하게, 가까운 곳에서 필요한 물건을 직거래로 구하고 싶다.
- **공통**: 낯선 사람과의 거래임에도 메시지를 주고받으며 신뢰를 쌓고 약속을 잡고 싶다.

### 2.2 v1 비대상

- 기업·사업자 판매자 (개인 간 거래만)
- 결제 시스템이 필요한 비대면 배송 거래
- 해외 사용자

### 2.3 핵심 사용자 여정

**UJ-1. 지수가 안 쓰는 스탠드 조명을 동네에 내놓는다.**

- **페르소나 + 맥락:** 지수, 20대 직장인. 이사 후 남은 가구를 처분하고 싶다.
- **진입 상태:** 앱 최초 실행, 비인증 상태.
- **흐름:**
  1. 앱 실행 → 구글 소셜 로그인 → 동네(지역) 드롭다운 선택.
  2. 하단 탭 글쓰기 버튼 탭 → 갤러리에서 조명 사진 5장 선택.
  3. 카테고리 "가구/인테리어" 선택, 제목·가격·설명 입력 → 등록.
  4. 피드에 상품 카드가 노출되고, 지수는 홈으로 돌아온다.
  5. 민준으로부터 채팅 메시지 수신 → 푸시 알림 → 앱 진입 → 채팅에서 거래 장소·시간 조율.
  6. 직거래 완료 후 상품 상태를 "판매완료"로 변경.
- **클라이맥스:** 상품이 피드에 올라가고 첫 채팅 알림을 받는 순간.
- **해소 상태:** 상품 상태 = 판매완료. 지수의 채팅 목록에 거래 내역 보존.
- **엣지 케이스:** 사진 업로드 중 네트워크 끊김 → 오류 토스트 표시, 이미 업로드된 사진 유지하고 재시도 가능.

---

**UJ-2. 민준이 동네 피드에서 조명을 발견하고 채팅을 건다.**

- **페르소나 + 맥락:** 민준, 30대 자취생. 저렴한 조명을 찾고 있다.
- **진입 상태:** 앱 실행, 비인증 상태.
- **흐름:**
  1. 앱 실행 → 로그인 없이 메인 피드 조회.
  2. 스크롤하며 탐색 → 지수의 조명 카드 발견 → 상품 상세 진입.
  3. 이미지 슬라이드·설명 확인, 하트(관심) 버튼 탭 → 로그인 유도 팝업.
  4. 구글 로그인 완료 → 관심 등록 처리.
  5. "채팅하기" 버튼 탭 → 채팅방 생성 → 첫 메시지 전송.
  6. 지수의 답장 실시간 수신 → 거래 장소·시간 확정.
- **클라이맥스:** 지수의 답장이 지연 없이 나타나는 순간.
- **해소 상태:** 채팅 목록에 대화 내역 보존, 거래 약속 완료.
- **엣지 케이스:** 이미 채팅 중인 상품에서 "채팅하기" 재탭 → 기존 채팅방으로 진입 (중복 생성 없음).

## 3. 용어 사전

| 용어 | 정의 |
|---|---|
| **상품(Product)** | 판매자가 등록한 중고 물품 1건. 상태(판매중/예약중/판매완료)를 가진다. |
| **판매자(Seller)** | 상품을 등록한 인증된 사용자. |
| **구매자(Buyer)** | 판매자의 상품에 채팅을 건 인증된 사용자. |
| **피드(Feed)** | 동네 기준으로 필터링된 최신 상품 목록 화면. |
| **채팅방(Chat Room)** | 상품 1건에 연결된 판매자-구매자 1:1 대화 공간. 상품당 구매자 1명당 1개 생성. |
| **동네(Neighborhood)** | 사용자가 드롭다운으로 선택한 지역 단위. 피드 필터링 기준. |
| **관심(Like)** | 구매자가 상품에 표시하는 하트. 관심 수는 피드 카드에 노출됨. |
| **판매 상태** | 상품의 현재 거래 단계: `판매중` → `예약중` → `판매완료`. |

## 4. 기능 요구사항

### 4.1 인증 (Auth)

**설명:** 소셜 로그인으로 진입 장벽을 낮추고, 비인증 사용자에게는 읽기 전용 접근만 허용한다. 실현: UJ-1, UJ-2.

**기능 요구사항:**

#### FR-1: 소셜 로그인

인증되지 않은 사용자는 구글 계정으로 로그인할 수 있다. 카카오 로그인은 FastAPI OAuth 연동 가능 여부 확인 후 추가한다(OQ-1).

**결과 (테스트 가능):**
- 로그인 성공 시 FastAPI가 JWT 액세스 토큰과 리프레시 토큰을 발급하고 Users 테이블에 upsert된다.
- 로그인 후 이전 시도한 화면(채팅하기, 글쓰기 등)으로 자동 복귀한다.

**Out of Scope:** 카카오 로그인은 OQ-1 해소 전까지 미구현.

#### FR-2: 비인증 접근 제어

비인증 사용자는 피드 조회와 상품 상세 조회만 가능하다.

**결과 (테스트 가능):**
- 채팅하기, 글쓰기, 관심 버튼 탭 시 로그인 유도 팝업 또는 로그인 화면으로 전환된다.
- 피드와 상세 화면은 로그인 없이 정상 렌더링된다.

#### FR-3: 세션 유지

앱을 다시 열 때 유효한 JWT가 있으면 로그인 화면을 건너뛴다.

**결과 (테스트 가능):**
- 저장된 JWT 액세스 토큰이 유효하면 자동 인증 상태 복원.
- 액세스 토큰 만료 시 리프레시 토큰으로 자동 갱신, 리프레시 토큰도 만료 시 로그인 화면으로 이동.

#### FR-4: 로그아웃

인증된 사용자는 설정 화면에서 로그아웃할 수 있다.

**결과 (테스트 가능):**
- 로그아웃 후 세션이 삭제되고 피드 화면(비인증)으로 이동한다.

---

### 4.2 동네 설정

**설명:** 위치 기반 서비스를 드롭다운 선택으로 단순화한다. 선택된 동네가 피드의 필터 기준이 된다. 실현: UJ-1.

#### FR-5: 동네 선택

인증된 사용자는 드롭다운에서 동네를 선택하고 저장할 수 있다.

**결과 (테스트 가능):**
- 선택 저장 후 피드가 해당 동네 기준으로 필터링된 상품만 표시한다.
- 동네 미설정 사용자가 글쓰기를 시도하면 동네 설정 화면으로 우선 안내한다.

#### FR-6: 동네 변경

사용자는 언제든지 동네를 변경할 수 있다.

**결과 (테스트 가능):**
- 변경 즉시 피드가 새 동네 기준으로 갱신된다.

---

### 4.3 메인 피드

**설명:** 앱의 진입점. 선택된 동네의 최신 상품을 무한 스크롤로 탐색한다. 비인증 사용자도 접근 가능. 실현: UJ-2.

#### FR-7: 상품 목록 조회

피드 화면은 현재 동네의 `판매중` 상품을 최신 등록 순으로 표시한다.

**결과 (테스트 가능):**
- 상품 카드에 썸네일(첫 번째 이미지)·제목·가격·등록 경과 시간·관심 수가 표시된다.
- `예약중` 상품은 카드에 "예약중" 배지를 표시하며 목록에 포함된다.
- `판매완료` 상품은 피드에서 제외된다.

#### FR-8: 무한 스크롤 페이지네이션

피드는 한 번에 20개씩 로드하며, 스크롤이 하단에 닿으면 다음 20개를 자동 로드한다.

**결과 (테스트 가능):**
- 20개 미만 남았을 때 추가 로드 없이 목록 끝을 안내한다.
- 로드 중 스켈레톤 UI가 표시된다.

#### FR-9: 피드 새로고침

피드 화면을 아래로 당기면(Pull-to-refresh) 목록이 최신 데이터로 갱신된다.

**결과 (테스트 가능):**
- 새로고침 완료 후 가장 최신 상품이 최상단에 노출된다.

#### FR-10: 상품 상세 진입

사용자는 피드 카드를 탭하여 상품 상세 화면으로 진입할 수 있다.

---

### 4.4 상품 상세

**설명:** 상품의 전체 정보를 보여주고, 구매자가 행동(채팅, 관심)을 취할 수 있는 화면. 실현: UJ-2.

#### FR-11: 상품 정보 표시

상품 상세 화면은 이미지 슬라이드·제목·가격·설명·카테고리·등록 시간·관심 수·판매 상태·판매자 닉네임을 표시한다.

**결과 (테스트 가능):**
- 이미지가 2장 이상이면 좌우 스와이프로 탐색 가능하다.
- 판매 상태가 `판매완료`인 경우 "채팅하기" 버튼이 비활성화된다.

#### FR-12: 관심 등록/해제

사용자는 하트 버튼을 탭하여 관심을 등록하거나 해제할 수 있다. 비인증 상태에서 탭하면 로그인 유도(FR-2).

**결과 (테스트 가능):**
- 관심 등록 시 하트가 채워지고 관심 수가 1 증가한다.
- 동일 사용자의 중복 관심 등록은 불가하다.

#### FR-13: 채팅하기

구매자는 "채팅하기" 버튼을 탭하여 해당 상품의 채팅방에 진입한다. 비인증 상태에서 탭하면 로그인 유도(FR-2).

**결과 (테스트 가능):**
- 기존 채팅방이 있으면 신규 생성 없이 기존 채팅방으로 진입한다. (UJ-2 엣지 케이스)
- 판매자가 본인 상품 상세에서 "채팅하기"를 탭하면 해당 상품의 채팅 목록으로 이동한다.

#### FR-14: 상품 수정 (판매자)

판매자는 본인 상품의 상세 화면에서 제목·가격·설명·판매 상태를 수정할 수 있다.

**결과 (테스트 가능):**
- 수정 저장 후 상세 화면이 즉시 반영된다.
- 타인의 상품에서는 수정 버튼이 노출되지 않는다.

#### FR-15: 상품 삭제 (판매자)

판매자는 본인 상품을 삭제할 수 있다.

**결과 (테스트 가능):**
- 삭제 확인 다이얼로그 후 삭제 처리되고 피드에서 즉시 제거된다.
- 연결된 채팅방은 보존되며 채팅 내 상품 정보에 "삭제된 상품" 표시.

---

### 4.5 상품 등록

**설명:** 판매자가 사진과 텍스트를 입력하여 상품을 등록하는 흐름. 카메라/갤러리 → 이미지 업로드 → DB 저장. 실현: UJ-1.

#### FR-16: 이미지 선택

판매자는 디바이스 카메라 또는 갤러리에서 사진을 선택할 수 있다. 최대 10장.

**결과 (테스트 가능):**
- OS 권한 요청 후 거부 시 권한 안내 화면으로 이동한다.
- 10장 초과 선택 시 경고 토스트를 표시하고 10장만 유지한다.
- 선택된 사진은 등록 전 미리보기로 표시된다.

#### FR-17: 이미지 업로드

등록 제출 시 선택된 이미지를 FastAPI 엔드포인트를 통해 파일 스토리지에 업로드한다. `[ASSUMPTION: A-1 — 스토리지 백엔드는 OQ-6에서 확정]`

**결과 (테스트 가능):**
- 업로드 전 클라이언트(Flutter)에서 이미지를 리사이즈하여 장당 최대 1MB로 압축한다. `[ASSUMPTION: A-1 — flutter_image_compress 라이브러리 사용]`
- 업로드 실패 시 실패한 이미지만 재시도 가능하다. (UJ-1 엣지 케이스)
- 업로드 진행률이 UI에 표시된다.

#### FR-18: 상품 정보 입력

판매자는 카테고리(드롭다운)·제목·가격·설명을 입력한다.

**결과 (테스트 가능):**
- 제목(필수, 최대 40자)·가격(필수, 숫자만)·카테고리(필수) 미입력 시 제출 버튼이 비활성화된다.
- 가격에 "무료나눔" 선택 옵션 제공. `[ASSUMPTION: 0원 입력으로 처리]`

**카테고리 목록:** 디지털기기 / 의류·잡화 / 가구·인테리어 / 생활가전 / 도서·티켓 / 스포츠·레저 / 기타

#### FR-19: 상품 등록 제출

판매자는 입력 완료 후 등록 버튼을 탭하여 상품을 DB에 저장한다.

**결과 (테스트 가능):**
- 등록 성공 시 피드로 이동하며 방금 등록한 상품이 최상단에 노출된다.
- 등록 중 네트워크 오류 시 오류 메시지를 표시하고 입력 내용을 보존한다.

---

### 4.6 1:1 실시간 채팅

**설명:** 구매자와 판매자 간 실시간 메시지 교환. FastAPI WebSocket으로 연결을 유지하며, 메시지는 PostgreSQL Messages 테이블에 영구 저장된다. 실현: UJ-1, UJ-2.

#### FR-20: 채팅방 생성

구매자가 "채팅하기"를 탭하면 해당 상품에 대한 채팅방이 생성된다.

**결과 (테스트 가능):**
- 동일 상품에 동일 구매자의 채팅방은 중복 생성되지 않는다.
- 채팅방 생성과 동시에 채팅 화면으로 진입한다.

#### FR-21: 메시지 전송

인증된 사용자는 텍스트 메시지를 전송할 수 있다.

**결과 (테스트 가능):**
- 전송된 메시지가 Messages 테이블에 저장된다.
- 전송 직후 본인 화면에 즉시 표시된다.
- 빈 메시지는 전송되지 않는다.

#### FR-22: 실시간 메시지 수신

채팅방 화면에 있는 사용자는 상대방의 메시지를 실시간으로 수신한다.

**결과 (테스트 가능):**
- FastAPI WebSocket 연결 기반으로 새 메시지 전송 시 2초 이내 화면에 표시된다.
- 새 메시지 수신 시 스크롤이 자동으로 최하단으로 이동한다.
- WebSocket 연결 끊김 시 자동 재연결 시도 후 실패 안내 토스트를 표시한다.

#### FR-23: 채팅 목록

인증된 사용자는 본인이 참여 중인 채팅방 목록을 볼 수 있다.

**결과 (테스트 가능):**
- 채팅방 목록에 상품 썸네일·상품 제목·마지막 메시지·시간·읽지 않은 메시지 수가 표시된다.
- 최근 메시지 기준 내림차순 정렬.

#### FR-24: 채팅 내 상품 정보

채팅 화면 상단에 연결된 상품의 썸네일·제목·가격·판매 상태가 표시된다.

**결과 (테스트 가능):**
- 상품 정보 탭 시 상품 상세 화면으로 이동한다.

---

### 4.7 푸시 알림

**설명:** 앱이 백그라운드이거나 종료된 상태에서 새 채팅 메시지를 사용자에게 알린다. Flutter 앱은 FCM(Firebase Cloud Messaging)을 통해 푸시 알림을 수신하며, FastAPI 백엔드가 메시지 저장 시 FCM으로 전송 요청을 보낸다. `[ASSUMPTION: A-3]`

#### FR-25: 채팅 알림

새 메시지가 수신되면 수신자에게 푸시 알림을 전송한다.

**결과 (테스트 가능):**
- 알림에 발신자 닉네임과 메시지 미리보기(최대 50자)가 포함된다.
- 알림 탭 시 해당 채팅방으로 딥링크 진입한다.
- 채팅방 화면이 활성화된 상태에서는 알림을 표시하지 않는다.

#### FR-26: 알림 권한 및 FCM 토큰 등록

앱 최초 실행 시 OS 푸시 알림 권한을 요청하고, 허용 시 FCM 디바이스 토큰을 FastAPI 서버에 등록한다.

**결과 (테스트 가능):**
- FCM 토큰이 Users 테이블에 저장되고 이후 알림 발송에 사용된다.
- 권한 거부 시 인앱 알림(배지)만으로 폴백 처리한다. `[ASSUMPTION: A-4]`
- 토큰 갱신 시(FCM 재발급) 서버에 자동 업데이트된다.

---

### 4.8 프로필 & 설정

**설명:** 사용자 닉네임은 상품 상세(FR-11)와 채팅(FR-25)에서 노출된다. 최초 로그인 시 닉네임을 설정하는 온보딩 단계와, 이후 수정 가능한 설정 화면이 필요하다. 실현: UJ-1.

#### FR-27: 닉네임 온보딩

소셜 로그인 최초 완료 후 닉네임 입력 화면을 표시한다.

**결과 (테스트 가능):**
- 닉네임(2~15자, 특수문자 제외) 미입력 시 완료 버튼이 비활성화된다.
- 완료 후 Users 테이블 nickname 컬럼에 저장되고 피드 화면으로 이동한다.
- 재로그인 시 닉네임이 이미 있으면 온보딩 화면을 건너뛴다.

#### FR-28: 프로필 편집

인증된 사용자는 설정 화면에서 닉네임과 프로필 사진을 변경할 수 있다.

**결과 (테스트 가능):**
- 변경 저장 후 해당 사용자의 상품 상세와 채팅방에 즉시 반영된다.
- 프로필 사진은 갤러리에서 선택하며 Supabase Storage에 업로드된다.

#### FR-29: 설정 화면

인증된 사용자는 설정 화면에서 로그아웃(FR-4)과 프로필 편집(FR-28)에 접근할 수 있다.

**결과 (테스트 가능):**
- 설정 화면은 하단 탭 또는 프로필 아이콘에서 진입 가능하다.

---

### 4.9 어드민 패널 (Next.js)

**설명:** 운영자가 서비스 상태를 모니터링하고 기본적인 콘텐츠·사용자 관리를 수행하는 내부 도구. FastAPI 동일 백엔드를 사용하되 어드민 전용 엔드포인트로 분리. 일반 사용자는 접근 불가.

#### FR-30: 어드민 인증

어드민 사용자는 이메일·비밀번호로 로그인하며, 역할(role = admin)이 부여된 계정만 접근 가능하다.

**결과 (테스트 가능):**
- 비어드민 계정으로 접근 시 403 반환 후 로그인 화면으로 리다이렉트된다.
- 어드민 JWT는 일반 사용자 JWT와 동일한 서버에서 발급하되 role 클레임이 `admin`으로 설정된다.

#### FR-31: 상품 관리

어드민은 전체 상품 목록을 조회하고 개별 상품을 삭제할 수 있다.

**결과 (테스트 가능):**
- 상품 목록은 등록일·상태·판매자로 필터링 가능하다.
- 삭제 시 확인 다이얼로그 후 처리되며 모바일 피드에서 즉시 제거된다.

#### FR-32: 사용자 관리

어드민은 가입된 사용자 목록을 조회하고 특정 사용자의 계정을 비활성화할 수 있다.

**결과 (테스트 가능):**
- 비활성화된 사용자는 로그인 시 "계정이 정지되었습니다" 오류를 수신한다.
- 비활성화된 사용자의 상품은 피드에서 숨김 처리된다.

#### FR-33: 대시보드

어드민 홈에 기본 운영 지표를 표시한다.

**결과 (테스트 가능):**
- 표시 항목: 총 가입자 수, 총 등록 상품 수, 오늘 신규 가입, 오늘 등록 상품, 활성 채팅방 수.
- 데이터는 페이지 진입 시 최신 값으로 로드된다.

## 7. 성공 지표

**Primary**

- **SM-1**: 판매자 여정(UJ-1) 전체 무결 동작률 — 상품 등록부터 판매완료 처리까지 오류 없이 완료되는 비율 ≥ 100%. 검증: FR-16~FR-19, FR-14.
- **SM-2**: 구매자 여정(UJ-2) 전체 무결 동작률 — 피드 진입부터 채팅 메시지 전송까지 오류 없이 완료되는 비율 ≥ 100%. 검증: FR-7~FR-13, FR-21.
- **SM-3**: 채팅 메시지 실시간 수신 지연 < 2초. 검증: FR-22.

**Secondary**

- **SM-4**: 메인 피드 초기 로딩 시간 < 3초 (3G 환경 기준). 검증: FR-7.
- **SM-5**: 앱 크래시 없는 세션 비율 ≥ 95%.

**Counter-metrics (최적화 금지)**

- **SM-C1**: 이미지 업로드 실패율 < 5%. — 압축 품질을 과도하게 낮춰 실패율을 줄이는 방식 금지. 검증: FR-17.
- **SM-C2**: 채팅 메시지 손실율 = 0%. — 실시간성을 높이기 위해 메시지 저장을 생략하는 방식 금지.

**Secondary (추가)**

- **SM-6**: 상품 등록 흐름 완료 시간 — 사진 선택부터 등록 완료까지 3분 이내 (정상 네트워크). 검증: FR-16~FR-19.

## 9. 오픈 질문

1. **카카오 로그인** — FastAPI에서 카카오 OAuth 2.0 연동 복잡도 사전 검증 필요. (구글은 표준 OIDC, 카카오는 별도 SDK 흐름)
2. **WebSocket 확장성** — FastAPI WebSocket을 단일 프로세스로 운영 시 동시 연결 한계 확인 필요. MVP 동시 사용자 규모 가정: 100명 이하. `[ASSUMPTION: A-6]`
3. **FCM 연동** — Firebase 프로젝트 생성 및 서비스 계정 키 발급 필요. Flutter 앱 google-services.json / GoogleService-Info.plist 설정 포함.
4. **동네 목록 데이터** — 전국 행정동 데이터를 어디서 가져와 어떻게 관리할지 미결. `[NOTE FOR PM: 피드 필터(FR-5, FR-7)와 UX 드롭다운 설계의 전제 조건 — 아키텍처 착수 전 반드시 확정 필요]`
5. **채팅 읽음 처리** — 읽지 않은 메시지 수 집계 방식 (DB 쿼리 vs. 클라이언트 계산) 아키텍처 결정 필요.
6. **파일 스토리지** — 이미지 저장소로 AWS S3, Cloudflare R2, MinIO(자체 호스팅) 중 선택 필요. 비용·운영 복잡도 비교 후 결정.
