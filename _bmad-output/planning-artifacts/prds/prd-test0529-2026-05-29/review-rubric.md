# PRD 품질 검토 — 중고거래 모바일 플랫폼 MVP

## 종합 판정

전반적으로 MVP 범위가 명확하고 UJ → FR → SM 연결 고리가 잘 유지된 성실한 PRD다. 가정 인덱스, 카운터 메트릭, 비목표 명시 등 구조적 습관이 좋다. 그러나 "동네" 정의의 실질적 미결, 프로필·사용자 신원 기능의 묵시적 범위 공백, 그리고 일부 성공 지표(≥ 100%)의 비현실성이 하위 설계 단계에서 재작업을 유발할 위험이 있다.

---

## 1. 의사결정 준비도 — adequate

FR과 UJ가 연결되어 있고, 오픈 질문 섹션이 미결 의사결정을 명시적으로 포착한다. 그러나 오픈 질문 §9의 항목 중 일부(특히 Q4 동네 목록, Q5 읽음 처리)는 UX 설계나 아키텍처 착수 전에 답이 나와야 하는 **착수 차단(blocker)** 사안인데, 우선순위나 결정 기한이 없다. 트레이드오프(예: Supabase Free tier 제약이 아키텍처 선택을 어떻게 바꾸는지)가 서술되지 않는다.

### 발견 사항

- **high** Q4 동네 목록 미결 (§9 Q4) — 드롭다운 UI를 설계하려면 동네 데이터 출처·구조가 확정되어야 한다. 행정동 3,500개 전체를 쓸지, 시/구 수준으로 축소할지에 따라 UX와 DB 스키마가 달라진다. *수정:* "MVP에서는 서울 25개 구를 고정 목록으로 하드코딩, v2에서 전국 행정동으로 확장" 같은 잠정 결정을 ASSUMPTION으로 등록하고 A-6으로 인덱싱.
- **high** Q5 읽음 처리 미결 (§9 Q5, FR-23) — 채팅 목록의 "읽지 않은 메시지 수" 표시(FR-23 결과)는 이미 명시되어 있으나 구현 방식이 오픈이다. 아키텍처 설계 첫날 결정되어야 할 사안. *수정:* MVP에서 사용할 방식을 ASSUMPTION으로 선택하고, 장단점 한 줄 메모 추가.
- **medium** 오픈 질문 우선순위·기한 없음 (§9 전체) — 5개 질문 모두 동일 무게로 나열되어 있다. *수정:* 각 항목에 "착수 전 필수 / UX 전 필수 / v1 이전 가능" 등 결정 타이밍 레이블 추가.

---

## 2. 실질 vs 형식 — adequate

대부분의 FR에 "결과(테스트 가능)" 항목이 있어 형식이 내용을 받쳐준다. 비목표 표(§5, §6.2)는 이유가 명시되어 형식적 나열을 넘는다. 단, §8 NFR 테이블 일부는 보일러플레이트 수준이다.

### 발견 사항

- **medium** NFR 접근성 항목이 최소 기준만 나열 (§8) — "44×44pt 터치 영역" 하나만 명시되어 있다. 소비자 모바일 앱에서 색상 대비, VoiceOver/TalkBack 지원 범위가 생략되면 UX 설계자가 임의로 해석하게 된다. *수정:* "MVP에서는 터치 영역 준수만 보장, WCAG 2.1 AA는 v2 목표" 식으로 의도적 제한임을 명시.
- **low** "오프라인: 데이터 캐싱 없음" 선언 (§8) — 캐싱 없음은 결정이지만, 오프라인 시 어떤 화면이 빈 상태로 남는지(피드, 채팅 모두?)에 대한 구체성이 없다. *수정:* "캐시 없음" 결정의 UX 영향(피드 = 빈 화면, 채팅 = 입력 불가)을 한 줄 서술.

---

## 3. 전략적 일관성 — strong

PRD에 명확한 테제가 있다: "결제·GPS 없이 핵심 거래 흐름이 동작함을 검증." 비목표가 이 테제를 강화하고, 모든 기능이 UJ-1/UJ-2 두 여정으로 수렴된다. 카운터 메트릭(SM-C1, SM-C2)이 최적화 방향을 명시적으로 제한하는 것은 동급 PRD 대비 두드러지게 좋은 점이다.

### 발견 사항

- **low** SM-1, SM-2 목표치 "≥ 100%" 비현실적 (§7) — E2E 오류 없는 완료율 100%는 MVP 품질 목표로 무의미하다(어떤 시스템도 달성 불가). *수정:* "≥ 95% (내부 QA 테스트 기준)" 또는 "0개의 P0 버그 상태에서 출시" 같은 현실적 표현으로 대체.
- **low** SM-5 크래시 없는 세션 95% — 측정 방법 미명시. *수정:* "Sentry/Crashlytics 기준" 등 측정 도구 한 줄 추가.

---

## 4. 완료 명확성 — strong

26개 FR 중 25개가 "결과(테스트 가능)" 항목을 포함하며, 대부분 QA 자동화 기준으로 전환 가능한 수준이다. 상태 전이(`판매중 → 예약중 → 판매완료`) 명시, 엣지 케이스 2건(네트워크 단절, 중복 채팅방) 포함이 특히 좋다.

### 발견 사항

- **medium** FR-10에 테스트 가능 결과 없음 (§4.3 FR-10) — "탭하여 상세 화면으로 진입" 서술만 있고 결과 항목이 없다. 화면 전환 애니메이션, 로딩 상태, 비인증 허용 여부가 명확하지 않다. *수정:* "비인증 상태에서도 상세 화면 진입 가능, 화면 전환 시 상품 ID가 URL/파라미터로 전달됨" 등 결과 항목 추가.
- **medium** 프로필 화면 정의 부재 — FR-4 로그아웃이 "설정 화면"을 참조하나 설정/프로필 화면은 FR로 정의되지 않았다. 사용자 닉네임(FR-11에서 노출됨)이 어디서 설정되는지 불명확. *수정:* 최소 FR로 "프로필(닉네임) 설정" 또는 "소셜 로그인 닉네임 자동 사용 [ASSUMPTION]"을 명시.
- **low** FR-14 수정 대상에서 카테고리·이미지 제외 — 제목·가격·설명·판매 상태만 수정 가능하다고 명시되어 있는데, 카테고리와 이미지 수정 불가 여부가 의도된 제한인지 누락인지 불명확. *수정:* "카테고리·이미지 수정은 MVP 제외 (비목표 명시 또는 §5에 추가)."

---

## 5. 범위 정직성 — adequate

§5 비목표와 §6.2 MVP 제외 테이블이 이유와 함께 제공되어 범위 정직성이 높다. ASSUMPTION 태그 5개가 §10 가정 인덱스와 일치한다. 그러나 사용자 프로필, 신고/차단 부재의 신뢰 리스크, 동네 데이터 출처가 묵시적 가정으로 남아 있다.

### 발견 사항

- **high** 사용자 프로필 기능 범위 공백 — 닉네임(FR-11)·설정 화면(FR-4) 참조가 있으나 프로필 조회·편집 FR이 없다. 이 기능이 MVP 포함인지 아닌지 미결. *수정:* §5 또는 §6.2에 "사용자 프로필 페이지 (닉네임 외) — MVP 제외" 명시하거나 최소 FR 추가.
- **medium** 신고/차단 부재의 신뢰 리스크 미서술 (§5, §6.2) — 비목표로만 나열되고 "소수 사용자라 운영 부담 최소화"로 이유가 설명된다. 그러나 악성 사용자 발생 시 운영 대응 방안(수동 삭제, 계정 정지 등)이 전혀 없다. *수정:* "MVP에서는 관리자 Supabase 대시보드 직접 조작으로 대응" 같은 운영 폴백을 한 줄 추가.
- **low** A-5 데이터 보존 30일 법적 미검토 명시 (§10) — 가정으로 등록은 되어 있으나 한국 개인정보보호법(PIPA)상 탈퇴 후 데이터 보존 기간 규정과의 충돌 가능성이 오픈 질문에 없다. *수정:* §9에 "Q6: 탈퇴 후 데이터 보존 기간 — PIPA 준수 여부 법무 확인 필요"로 이동.

---

## 6. 하위 사용성 — adequate

용어 사전(§3)이 8개 핵심 용어를 정의하고 PRD 전반에서 일관되게 사용된다. UJ에 이름 있는 페르소나(지수, 민준)가 등장한다. FR-ID(FR-1~FR-26)와 SM-ID(SM-1~SM-5, SM-C1~SM-C2)가 연속적이다. 그러나 UJ-ID가 FR 결과 항목에서 역참조되지 않는 경우가 있고, 아키텍처 설계자가 필요로 할 데이터 모델 힌트가 부족하다.

### 발견 사항

- **medium** 데이터 모델 힌트 부족 — 아키텍처 설계자가 테이블 구조를 추론해야 한다. `Users`, `Messages` 테이블은 FR 내에서 언급되나 `Products`, `ChatRooms`, `Likes` 엔티티 명칭이 비일관적으로 노출된다. *수정:* §3 용어 사전 또는 별도 §3.1 엔티티 요약에서 핵심 테이블명·관계를 1~2줄로 명시 (스키마 아님, 명칭 합의 수준).
- **medium** FR-25 알림 구현 가정(A-3)과 §9 Q3 중복 — A-3은 "Expo + Edge Function"으로 구현을 결정한 ASSUMPTION이고, Q3는 같은 사안을 미결 질문으로 올려놓았다. *수정:* Q3를 가정으로 승격하거나, A-3을 삭제하고 Q3에서 결정하도록 통일.
- **low** UJ-1, UJ-2 엣지 케이스와 FR 상호참조 — UJ-1 엣지 케이스(네트워크 끊김)는 FR-17에서 참조되나, UJ-2 엣지 케이스(중복 채팅방 방지)는 FR-13과 FR-20에서 각각 처리된다. 독자가 UJ에서 FR을 쉽게 추적할 수 있도록 UJ 흐름 단계에 FR-ID 인라인 참조 추가를 권장.

---

## 7. 형태 적합성 — strong

React Native(Expo) + Supabase 스택 선택이 명시되고, 플랫폼(iOS 14+/Android 10+), 실시간 기술(Supabase Realtime), 이미지 처리 전략(expo-image-manipulator)이 구체적으로 명시되어 있다. UJ가 모바일 터치 인터랙션(탭, 스와이프, 풀투리프레시)을 직접 서술한다. 이 PRD는 UX 설계자와 아키텍트가 독립적으로 착수할 수 있는 수준의 정보를 제공한다.

### 발견 사항

- **low** 딥링크 스킴 미정의 (FR-25) — "알림 탭 시 해당 채팅방으로 딥링크 진입"이 명시되나 Expo의 딥링크 스킴(`exp://`, 커스텀 스킴, Universal Link) 방식이 미결. *수정:* ASSUMPTION으로 "MVP에서는 Expo Go 호환 커스텀 스킴 사용" 등 최소 방향성 기록.
- **low** 탭 네비게이션 구조 암시적 — UJ-1에서 "하단 탭 글쓰기 버튼"이 언급되나 전체 탭 구조(홈/채팅/프로필 등)가 명시되지 않는다. *수정:* §2.3 또는 §4 서두에 "하단 탭: 홈(피드) / 글쓰기 / 채팅 목록 / 마이페이지" 구조를 한 줄로 명시.

## 기계적 검토

**용어 일관성**
- "상품(Product)"은 §3 정의와 일치하나, FR-23에서 "상품 썸네일·상품 제목"으로 일관되게 사용됨. 이상 없음.
- "채팅방(Chat Room)"은 §3 정의와 FR 전반에서 일관됨.
- FR-21에서 "Messages 테이블", FR-1에서 "Users 테이블" 등 DB 테이블명이 FR 내에 산발적으로 등장하나 §3 용어 사전에는 없음 — 경미한 드리프트.

**ID 연속성**
- FR-1 ~ FR-26: 연속적, 누락 없음.
- SM-1 ~ SM-5, SM-C1 ~ SM-C2: 연속적, 누락 없음.
- UJ-1 ~ UJ-2: 연속적.
- A-1 ~ A-5: §10 인덱스와 본문 `[ASSUMPTION]` 태그 5개 일치 확인.
  - FR-17에 A-1, FR-18에 A-2, FR-25에 A-3, FR-26에 A-4, §8에 A-5. 모두 매칭.
  - **불일치**: FR-25 설명 단락의 `[ASSUMPTION: Expo Notifications + Supabase Edge Function으로 구현]` 태그가 A-3으로 인덱스됨. 그러나 §9 Q3는 동일 사안을 오픈 질문으로 처리 — 논리적 충돌 (§6 발견 사항 medium 항목과 동일).

**가정 인덱스 일치 여부**
- A-5 법적 미검토 메모가 §10 인덱스에 명시("법적 요구 미검토")되어 있어 리스크 인식은 적절히 포착됨.
- §9 오픈 질문과 §10 가정 인덱스 간 경계가 다소 모호함: Q1(카카오 로그인)은 가정으로 전환 가능한 수준이나 여전히 질문으로만 남아 있음.
